**Key Learnings:**
- **LocalStorage Limitations:**
  - Can only store strings (need JSON.stringify for objects/arrays)
  - Has a size limit (~5-10MB depending on browser)
  - Synchronous operations (can block UI for large data)
  - Cleared when user clears browser data
  
- **Event Delegation Benefits:**
  - Attach one listener instead of many
  - Works for dynamically added elements
  - Better memory performance
  - Cleaner code structure

- **Best Practices:**
  - Always parse with try-catch (localStorage can be corrupted)
  - Use meaningful key names
  - Consider using a namespace for your app's storage keys
  - Clear old data to avoid accumulation

**Code Snippet - LocalStorage Helper Functions:**
```javascript
// Reusable localStorage utilities I created
const storage = {
  save: (key, data) => {
    try {
      localStorage.setItem(key, JSON.stringify(data));
      return true;
    } catch (e) {
      console.error('Failed to save to localStorage', e);
      return false;
    }
  },
  
  load: (key, defaultValue = null) => {
    try {
      const data = localStorage.getItem(key);
      return data ? JSON.parse(data) : defaultValue;
    } catch (e) {
      console.error('Failed to load from localStorage', e);
      return defaultValue;
    }
  },
  
  remove: (key) => {
    localStorage.removeItem(key);
  },
  
  clear: () => {
    localStorage.clear();
  }
};
```

**Deep Dive - State Management Concepts:**

**What is State?**
State is the data that drives your application's UI. When state changes, the UI should automatically update to reflect those changes.

**Why Separate State from UI?**
- **Single Source of Truth:** One place for all data
- **Predictability:** Know exactly what causes UI updates
- **Debugging:** Easier to track state changes
- **Testing:** Can test state logic independently of UI
- **Scalability:** Easier to add features and manage complexity

**My State Management Implementation:**
```javascript
class SimpleStore {
  constructor(initialState = {}) {
    this.state = initialState;
    this.listeners = [];
  }
  
  // Get current state
  getState() {
    return this.state;
  }
  
  // Update state and notify listeners
  setState(newState) {
    this.state = { ...this.state, ...newState };
    this.notify();
  }
  
  // Subscribe to state changes
  subscribe(listener) {
    this.listeners.push(listener);
    
    // Return unsubscribe function
    return () => {
      this.listeners = this.listeners.filter(l => l !== listener);
    };
  }
  
  // Notify all listeners of state change
  notify() {
    this.listeners.forEach(listener => listener(this.state));
  }
}

// Usage example:
const store = new SimpleStore({ count: 0 });

// Subscribe to changes
store.subscribe(state => {
  console.log('State changed:', state);
  document.getElementById('count').textContent = state.count;
});

// Update state
store.setState({ count: store.getState().count + 1 });
```

**Key Learnings:**
- **Immutability is Crucial:** Always create new objects instead of mutating existing ones
- **Observer Pattern:** The foundation of reactive systems (used in React, Vue, Angular)
- **Separation of Concerns:** State logic should be separate from UI logic
- **State Flow:** State changes â†’ Notify subscribers â†’ Update UI
- **Debugging:** Console.logging state changes helps tremendously

**Real-World Application:**
This is essentially how frameworks like React (useState), Vue (reactive), and Angular (RxJS) work under the hood. Building this from scratch helped me understand:
- Why React hooks work the way they do
- How component re-rendering is triggered
- Why we need state management libraries for larger apps

**Challenges Faced:**
- Initially mutated state directly instead of creating new objects
- Forgot to notify listeners after state updates
- Struggled with deciding what should be state vs derived data
- Had issues with stale closures in event handlers

**Breakthrough Moment:**
When I realized that state management is just the observer pattern + immutability, everything clicked! This is the same pattern used everywhere in modern frontend.

**Resources Used:**
- JavaScript Design Patterns - Observer Pattern
- Redux documentation (for conceptual understanding)
- "You Don't Need Redux" blog post
- Built my own mini-library by following the mental model


**Comments:**
```javascript

// âŒ Bad: Stating the obvious
// This function adds two numbers
```

```javascript
// âœ… Good: Graceful degradation
try {
  const data = JSON.parse(localStorage.getItem('data'));

const data = JSON.parse(localStorage.getItem('data'));
```

- Always use JSON.parse when retrieving
- Add try-catch for error handling

**Lesson:** LocalStorage only stores strings

### Debugging Strategies

**1. Console Everything:**
```javascript
console.log('Before update:', state);
updateState(newData);
console.log('After update:', state);
```

**2. Use Debugger:**
```javascript
function complexFunction() {
  debugger; // Pause here
  // investigate variables
}
```

**3. Rubber Duck Debugging:**
- Explain code line-by-line to imaginary duck
- Often find the bug while explaining

**4. Binary Search Debugging:**
- Comment out half the code
- Find which half has the bug
- Repeat until found
